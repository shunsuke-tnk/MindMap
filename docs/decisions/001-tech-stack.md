# ADR 001: 技術スタックの選定

## ステータス

承認済み

## 背景

チームの会議・ブレストで使うリアルタイム共同編集マインドマップツールを開発するにあたり、
フレームワーク、描画ライブラリ、リアルタイム同期基盤を選定する必要がある。
ユーザーは特定のチーム（5〜10人）で、PCから利用する。

## 決定

以下の技術スタックを採用する:

- **フレームワーク**: Next.js 15（App Router）
- **スタイリング**: Tailwind CSS
- **言語**: TypeScript（strict mode）
- **ノード描画**: @xyflow/react（React Flow v12）
- **レイアウト計算**: d3-hierarchy + d3-flextree
- **リアルタイム同期**: Liveblocks
- **パッケージマネージャ**: npm

## 理由

### Next.js
ページ遷移・API通信・SSR が標準で備わっており、MVP を素早く構築できる。
App Router により、動的ルーティング（`/map/[roomId]`）が簡潔に実装できる。

### React Flow (@xyflow/react)
ノードのドラッグ、パン、ズーム、選択、接続などの基本操作が標準で提供される。
カスタムノード/エッジで XMind 風のデザインを実現でき、車輪の再発明を避けられる。

### d3-flextree
標準の `d3.tree()` は全ノード同サイズ前提だが、`d3-flextree` はノードごとに異なるサイズを指定可能。
テキスト量が異なるマインドマップノードのレイアウトに適している。

### Liveblocks
リアルタイム同期（CRDT）、プレゼンス（カーソル共有）、永続化がすべてマネージド。
自前で WebSocket サーバーを運用する必要がなく、MVP に最速で到達できる。
無料枠（月500ルーム）で小規模チーム利用に十分。

## 検討した代替案

| 代替案 | 不採用の理由 |
|--------|------------|
| Yjs + y-websocket | WebSocket サーバーの運用が必要。MVP 工数が 1.6〜1.8 倍に増加 |
| Supabase Realtime + カスタムSVG | SVG 描画エンジンの自前実装が大きな工数。CRDT なしでコンフリクト解決が困難 |

## 影響

- Next.js App Router 規約に従ったディレクトリ構成にする
- Liveblocks Storage を Single Source of Truth とし、React Flow の状態と双方向バインディングする
- 将来 Liveblocks に不満が出た場合、同期レイヤーのみ Yjs に差し替える移行パスを確保する
